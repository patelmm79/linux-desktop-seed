# Daily Session Notes — 2026-04-10

## dev-nexus WIF Setup — Full Day Fix

### Problem
GitHub Actions `terraform-apply.yml` workflow was failing on WIF (Workload Identity Federation) setup. Multiple root causes accumulated over prior sessions.

### What Was Broken

**1. Workflow YAML corruption**
- "Terraform WIF-only apply" step had duplicate `run:` key (line 267)
- `-var-file` passed to `terraform apply tfplan` (plan already has vars baked in)
- `wif_only` conditions wrong — CLI passed string `"true"`, not boolean `true`
- Init/Plan/force-unlock all ran when `wif_only=true` (should skip)

**2. Terraform state out of sync with GCP**
- GCS bucket, Secret Manager secrets, WIF resources existed in GCP but not in terraform state
- Ran `terraform-import.yml` to sync — bucket import failed (wrong bucket name hardcoded)
- Fixed bucket name in import workflow

**3. `google_service_account_iam_member` broken for principalSet**
- Known Hashicorp bug since 2022: `principalSet://` format rejected by GCP IAM API
- GitHub issue: #12446 — workaround was using `.id` instead of `.workload_identity_pool_id`
- Even with correct path format, still got "unknown type" errors
- **Root cause**: terraform mangled the principalSet identifier when calling GCP IAM API

### Solution Implemented

**Workflow changes (`terraform-apply.yml`):**
- Removed broken `google_service_account_iam_member` resources from terraform
- Added gcloud binding step in GitHub Actions (self-healing, idempotent)
- Fixed wif_only condition logic (string-aware)
- Changed full apply to direct apply (no tfplan dependency)
- Added refresh-only import step for WIF-only path

**Doc changes (`docs/github-actions-wif-setup.md`):**
- Rewrote entirely to use gcloud commands instead of terraform for WIF→SA bindings
- Added single copy-pasteable bash script (steps 1-6 in one block)
- Added three repo access modes (Option A: single repo, B: org-wide, C: multi-repo)
- Added error collection + SUCCESS/FAILED summary at end

### Bugs Found and Fixed in the Script

| Bug | Symptom | Fix |
|-----|---------|-----|
| `create-github` not recognized | gcloud 564 uses `create-oidc` | `--issuer-uri="https://token.actions.githubusercontent.com"` |
| Missing `/` in principalSet URL | `iam.googleapis.comprojects/...` | `iam.googleapis.com/${POOL}` |
| `principalSet://` for pool-level binding | "unknown type" | Pool-level needs `principal://`, not `principalSet://` |
| SA creation failed (already exists) | Error exit | `|| true` + `run()` helper |

### Key Findings

- **`principal://`** — pool-level binding (org-wide access), NOT `principalSet://`
- **`principalSet://`** — attribute-based binding (per-repo), used with `/attribute.repository/...`
- `gcloud` handles both formats correctly; terraform's `google_service_account_iam_member` does not
- `create-github` deprecated in newer gcloud — use `create-oidc --issuer-uri="https://token.actions.githubusercontent.com"`
- GitHub Actions self-healing step means bindings are re-applied on every deploy
