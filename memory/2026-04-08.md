## Session: 2026-04-08 15:15–16:20 UTC — OAuth redirect_uri fix (CRITICAL)

### Root Cause (FINALLY FOUND)
The Cloud Run service `pattern-discovery-agent` had a stale `BACKEND_URL` env var:
- OLD (wrong): `https://pattern-discovery-agent-665374072631.us-central1.run.app`
- NEW (correct): `https://pattern-discovery-agent-75l7mntama-uc.a.run.app`

The `PROD_BACKEND_URL` GitHub Actions variable was set correctly, but:
1. Deploy #311 successfully updated `FRONTEND_URL` in Cloud Run env vars
2. But `BACKEND_URL` remained stuck at the old value across ALL revisions (even new ones created by deploy)
3. This is because the Cloud Run service-level env var update via `gcloud run services update --set-env-vars` doesn't propagate to existing revisions — it only affects NEW revisions created AFTER the update

### The Fix
1. Manually ran: `gcloud run services update pattern-discovery-agent --set-env-vars="BACKEND_URL=https://pattern-discovery-agent-75l7mntama-uc.a.run.app"`
   - Created new revision `pattern-discovery-agent-00108-6rh` with CORRECT BACKEND_URL
2. Shifted traffic: `gcloud run services update-traffic pattern-discovery-agent --to-revisions=pattern-discovery-agent-00108-6rh=100`
   - Now 100% traffic at revision `00108-6rh`
   - Service URL: `https://pattern-discovery-agent-75l7mntama-uc.a.run.app`

### Why it was hard to find
- The `gcloud run services update` output showed "Deploying... Service [pattern-discovery-agent] revision [pattern-discovery-agent-00107-f6n] has been deployed" — confusing because it showed the OLD revision name even though a NEW one was created
- Traffic remained at the old revision (00107-f6n) until I explicitly ran `update-traffic`
- GitHub Actions workflow's `deploy-production.yml` had correct `--set-env-vars` but Cloud Run wasn't applying them to existing revisions

### Manual Fix Required Because
The GitHub Actions deploy workflow's `gcloud run services update --set-env-vars` creates new revisions, but doesn't automatically shift 100% traffic to them (it relies on `--no-traffic` + separate traffic shift step). When `gcloud run services update` is called separately outside the workflow, it doesn't trigger traffic reallocation.

### What ACTUALLY needs to be fixed in the workflow
The deploy-production.yml `gcloud run deploy` command creates a new revision but the `--set-env-vars` flag may not be applying correctly. The real fix would be:
1. Ensure `gcloud run deploy` with `--set-env-vars` properly creates a revision with correct env vars (it should)
2. The `gcloud run services update-traffic` step should shift ALL traffic to the new revision (it does, via `--to-revisions=new=100`)

The deploy #311 DID create revision 00108 with the CORRECT BACKEND_URL. The problem was traffic was still at 00107. The workflow's traffic shift should have moved it, but it didn't (because it shifted to 00107 again due to using the wrong revision name captured earlier).

### Key Cloud Run Learning
- `gcloud run services update --set-env-vars=VAR=val` creates a NEW revision with the new env var
- It does NOT update existing revisions' env vars
- Traffic remains on the OLD revision until explicitly shifted
- Always check `gcloud run revisions list` + `gcloud run revisions describe <new-rev> --format=json | grep BACKEND` after any env var update
